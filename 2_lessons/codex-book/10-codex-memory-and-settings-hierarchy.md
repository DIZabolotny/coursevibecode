# Глава 10. Как у Codex устроены память, настройки и уровни ограничений

**Для кого эта глава**

Эта глава написана для человека, который:

- уже видел `AGENTS.md` и `config.toml`, но не до конца понимает разницу;
- чувствует, что в Codex есть несколько уровней управления, но путает их;
- хочет навести порядок без лишней инженерной экзотики;
- хочет понимать, где заканчиваются инструкции и начинаются реальные ограничения.

## Зачем нужна эта глава

Когда человек впервые начинает работать с Codex глубже, у него быстро возникает путаница.

Потому что рядом существуют сразу несколько слоев:

- `AGENTS.md`;
- `.codex/config.toml`;
- `.rules`;
- `requirements.toml`.

Если их смешать, система становится хаотичной.

Поэтому здесь задача простая:
дать читателю ясную рабочую карту.

## Что вы сможете после этой главы

После этой главы вы сможете:

- понять, где в Codex живет память проекта;
- понять, где живут реальные настройки клиента;
- понять, зачем нужны `.rules`;
- понять, когда начинается организационный уровень;
- выбрать простую архитектуру без путаницы.

## Главная мысль главы

В Codex нет одного "волшебного файла".

Есть несколько разных механизмов:

- `AGENTS.md` хранит инструкции;
- `.codex/config.toml` хранит настройки;
- `.rules` управляет внешними командами вне sandbox;
- `requirements.toml` задает организационные барьеры.

Если говорить совсем по-человечески:

- `AGENTS.md` отвечает за "как здесь принято работать";
- `config.toml` отвечает за "в каком режиме вообще стартует агент";
- `.rules` отвечает за "что делать с командами вне sandbox";
- `requirements.toml` отвечает за "что в компании уже нельзя обойти".

## 1. Где в Codex живет память проекта

В Codex роль постоянной памяти играет `AGENTS.md`.

По официальной документации OpenAI Codex читает `AGENTS.md` до начала работы.

### Какие бывают уровни `AGENTS.md`

1. Личный уровень

Это `~/.codex/AGENTS.md` или `~/.codex/AGENTS.override.md`.

Здесь живут ваши общие привычки:

- отвечать кратко;
- сначала показывать план;
- после правок запускать проверку.

2. Проектный уровень

Это `AGENTS.md` в корне репозитория.

Здесь живет память конкретного проекта:

- для кого он;
- как устроены файлы;
- что нельзя менять;
- что считается хорошим результатом.

3. Локальный уровень подпапки

Если в проекте есть подпапка со своей логикой, там можно положить еще один `AGENTS.md`.

Чем ближе файл к текущей рабочей папке, тем он важнее.

### Что важно понять

`AGENTS.md` это не техническая конфигурация.

Это именно инструкция для агента.

Туда хорошо класть:

- цель проекта;
- аудиторию;
- стиль;
- definition of done;
- рабочие ограничения.

Туда плохо класть:

- секреты;
- токены;
- хаотические заметки "на всякий случай";
- технические параметры sandbox.

## 2. Где в Codex живут реальные настройки

Для этого у Codex есть `.codex/config.toml`.

Это уже не "памятка", а реальная конфигурация.

Через нее задают:

- модель;
- sandbox;
- approval policy;
- MCP;
- project defaults;
- часть поведения среды.

### Какие бывают уровни `config.toml`

По документации приоритет устроен так:

1. CLI flags и `--config`
2. profile
3. проектные `.codex/config.toml` от корня вниз к текущей папке
4. `~/.codex/config.toml`
5. системный `config.toml`
6. встроенные defaults

### Очень важный нюанс

Проектные `.codex/config.toml` загружаются только если проект trusted.

Если проект недоверенный, Codex их игнорирует.

Это важная защита.

## 3. Зачем нужны `.rules`

Вот здесь чаще всего и случается путаница.

В Codex `.rules` это не "правила стиля текста".

В Codex `.rules` нужны для другого:
управлять тем, как Codex запускает команды вне sandbox.

Через `prefix_rule()` вы можете сказать:

- `allow`
- `prompt`
- `forbidden`

То есть:

- разрешить;
- спрашивать;
- запрещать.

### Когда это нужно

Например:

- вы хотите, чтобы внешняя команда всегда спрашивала подтверждение;
- вы хотите запретить опасный класс команд;
- вы хотите, чтобы определенный префикс был разрешен без лишнего трения.

### Что важно помнить

Если несколько правил совпали, выигрывает более жесткое решение:

`forbidden` > `prompt` > `allow`

## Отдельный риск: почему нельзя бездумно принимать Smart approvals

Идеология здесь очень похожа на Claude Code, но механизм другой.

В Codex allow list обычно пишется в:

`~/.codex/rules/default.rules`

Кроме того, Smart approvals может предложить вам готовый `prefix_rule()`.

И вот здесь возникает риск:

- если команда уже содержит логины, IP, пути к ключам, токены или connection strings;
- а человек просто принимает предложенное правило;
- тогда чувствительные детали могут оказаться в локальных rules или даже в project `.codex/rules/*.rules`.

Это особенно важно для терминального Codex, потому что там люди часто работают с:

- `ssh`;
- `curl`;
- cloud CLI;
- shell-обертками;
- командами к базам данных.

### Безопаснее делать так

1. Держать реальные секреты отдельно, например в `.production-credentials`.
2. Не принимать allow list в слишком конкретном виде.
3. Оставлять в rules только общий префикс действия или имя безопасной wrapper-команды.
4. Иногда вручную проверять `~/.codex/rules/default.rules`.
5. Никогда не коммитить project `.codex/rules/*.rules` с чувствительными аргументами.

## 4. Как связаны sandbox и approvals

По официальной документации OpenAI у Codex есть два главных рычага безопасности:

- `sandbox_mode`
- `approval_policy`

Простыми словами:

- `sandbox_mode` отвечает за физические границы;
- `approval_policy` отвечает за моменты согласования.

### Основные sandbox modes

- `read-only`
- `workspace-write`
- `danger-full-access`

### Основные approval policies

- `untrusted`
- `on-request`
- `never`

### Самая полезная бытовая формула

Если сомневаетесь, идите по лестнице:

1. сначала чтение;
2. потом работа внутри проекта;
3. потом только при явной необходимости более широкий доступ.

Важно:

- `--full-auto` это не полный беспредел;
- по docs это `workspace-write` + `on-request`.

А реально опасный режим это уже `danger-full-access` вместе с `never`.

## 5. Где начинается организационный уровень

Если проект личный, вам обычно хватает:

- `AGENTS.md`
- `.codex/config.toml`
- `.rules`

Но в компании может появиться еще один уровень:

- `requirements.toml`
- `managed_config.toml`

### `requirements.toml`

Это слой жестких ограничений.

Он нужен, когда организация хочет сказать:

- нельзя full access;
- нельзя certain MCP;
- нельзя определенные approval modes.

### `managed_config.toml`

Это слой managed defaults.

Он не столько запрещает, сколько задает стартовые значения при запуске.

## 6. Самая простая рабочая схема

Если не хочется усложнять, используйте такую архитектуру.

### Личный слой

`~/.codex/AGENTS.md`

Туда:

- ваши общие привычки;
- стиль работы;
- любимый формат ответа.

`~/.codex/config.toml`

Туда:

- ваши глобальные настройки;
- любимая модель;
- ваш обычный sandbox.

### Слой проекта

`project-root/AGENTS.md`

Туда:

- смысл проекта;
- аудитория;
- правила работы;
- definition of done.

`project-root/.codex/config.toml`

Туда:

- безопасные defaults проекта;
- MCP проекта;
- project-specific runtime settings.

### Локальный слой подпапки

`subdir/AGENTS.md`

Туда:

- только локальные уточнения этой зоны.

### Слой исключений

`.codex/rules/*.rules`

Туда:

- правила про конкретные внешние команды.

## 7. Короткий playbook по кейсам

### Кейс 1. Я хочу, чтобы Codex во всех проектах работал в моем стиле

Решение:
`~/.codex/AGENTS.md`

### Кейс 2. Я хочу, чтобы один репозиторий запускался в более безопасном режиме

Решение:
`project-root/.codex/config.toml`

### Кейс 3. Я хочу отдельные правила только для одной подпапки

Решение:
`subdir/AGENTS.md`

### Кейс 4. Я хочу, чтобы внешняя команда всегда спрашивала подтверждение

Решение:
`.codex/rules/*.rules`

### Отдельный кейс безопасности. Я не хочу, чтобы allow list стал складом чувствительных команд

Решение:

1. Открыть `~/.codex/rules/default.rules`.
2. Если в проекте есть shared rules, открыть и `.codex/rules/*.rules`.
3. Найти, нет ли там токенов, логинов, IP, портов или путей к ключам.
4. Оставить в rules только общие префиксы действий или безопасные wrapper-команды.

### Кейс 5. Я хочу запретить full access на уровне организации

Решение:
`requirements.toml`

### Кейс 6. Я хочу на один запуск временно включить другой режим

Решение:
CLI flags или `--config`, а не переписывание всей архитектуры проекта.

## 8. Что не путать

Не путайте:

- `AGENTS.md` и `config.toml`;
- `.rules` и текстовые инструкции;
- проектный default и организационный запрет;
- временный override и постоянную политику.

И отдельно:

- не превращайте `~/.codex/rules/default.rules` в хранилище чувствительных shell-команд.

Если правило звучит как:

- "пиши проще" -> `AGENTS.md`
- "работай в workspace-write" -> `config.toml`
- "на эту команду спрашивай" -> `.rules`
- "всем нельзя full access" -> `requirements.toml`

## 9. Глоссарий

**`AGENTS.md`**
Файл постоянных инструкций для Codex.

**`.codex/config.toml`**
Конфигурационный файл Codex.

**`.rules`**
Файл правил для внешних команд вне sandbox.

**`~/.codex/rules/default.rules`**
Пользовательский allow list Codex.

**`sandbox_mode`**
Техническая граница доступа агента.

**`approval_policy`**
Граница согласования действий.

**`requirements.toml`**
Организационный слой жестких ограничений.

**`managed_config.toml`**
Организационный слой стартовых значений.
