# README — `setting.json`

## Что это

`setting.json` — это файл локальной конфигурации разрешений для Claude Code / Cowork, который определяет, какие действия агенту разрешено выполнять без дополнительных подтверждений.

В текущем виде файл переводит систему в очень свободный режим работы:
- включает `bypassPermissions` как режим по умолчанию;
- явно разрешает файловые операции;
- разрешает shell-команды;
- разрешает web access;
- разрешает вызовы агентных инструментов;
- разрешает интеграции с Claude in Chrome и Cowork MCP.

Иными словами, этот файл нужен для того, чтобы агент мог работать автономно и не останавливался на каждом действии.

---

## Содержимое файла

Конфигурация задаёт блок `permissions`, внутри которого:

- `defaultMode: "bypassPermissions"`  
  означает, что система по умолчанию не требует ручного подтверждения для разрешённых действий;

- `allow: [...]`  
  содержит список разрешённых capability-patterns.

В список разрешений включены:

- `Bash(*)`
- `Read(*)`
- `Write(*)`
- `Edit(*)`
- `Glob(*)`
- `Grep(*)`
- `WebSearch`
- `WebFetch(*)`
- `Agent(*)`
- `mcp__Claude_in_Chrome__*`
- `mcp__cowork__*`

---

## Для чего нужен этот файл

Файл нужен, когда вы хотите, чтобы агент мог сам:

- читать и писать файлы;
- редактировать структуру проекта;
- искать данные по диску;
- выполнять shell-команды;
- обращаться к вебу;
- управлять браузерными действиями через интеграции;
- вызывать другие агентные инструменты.

Без такого файла автономный pipeline либо не заработает, либо будет постоянно требовать ручного подтверждения.

---

## Почему он важен

Этот файл — ключевой элемент автономии.

`CLAUDE.md` задаёт **что делать**.  
`setting.json` задаёт **что разрешено делать**.

Без него агент знает методологию, но не имеет достаточной свободы для исполнения.  
С ним агент получает операционные права на выполнение workflow.

---

## Когда использовать

Используйте `setting.json`, если ваш сценарий требует:

- длинной автономной работы;
- массового создания и редактирования файлов;
- веб-навигации;
- доступа к браузерным интеграциям;
- минимизации ручных approval-паузы.

---

## Что означают разрешения

### `Bash(*)`
Разрешает shell-команды. Это нужно для:
- перемещения файлов;
- создания директорий;
- пакетной обработки;
- запуска утилит.

### `Read(*)`
Разрешает чтение файлов.

### `Write(*)`
Разрешает создание и запись файлов.

### `Edit(*)`
Разрешает изменение существующих файлов.

### `Glob(*)`
Позволяет искать файлы по шаблонам.

### `Grep(*)`
Позволяет искать текст по содержимому файлов.

### `WebSearch`
Разрешает веб-поиск.

### `WebFetch(*)`
Разрешает загрузку содержимого веб-страниц.

### `Agent(*)`
Разрешает вызовы агентных инструментов и подзадач.

### `mcp__Claude_in_Chrome__*`
Даёт доступ к browser automation через Claude in Chrome.

### `mcp__cowork__*`
Даёт доступ к Cowork-интеграциям.

## Плюсы такой конфигурации

### Максимум автономии
Агент может выполнять длинные цепочки действий без постоянных остановок.

### Подходит для extraction workflows
Особенно хорошо работает там, где много однотипных шагов:
- открыть,
- прочитать,
- извлечь,
- сохранить,
- перейти дальше.

### Уменьшает микроменеджмент
Пользователю не нужно вручную подтверждать каждую мелочь.

---

## Основной риск

Главный риск в том, что это **очень широкий уровень доступа**.

`bypassPermissions` + разрешение на `Bash(*)`, `Write(*)`, `Edit(*)`, web access и browser MCP означает, что агент получает почти полный operational freedom в рамках среды.

Это удобно, но требует:
- доверия к среде;
- понимания, где агент работает;
- контроля над тем, какие папки доступны;
- аккуратности с чувствительными файлами.

---

## Рекомендации по безопасности

### 1. Используйте отдельную рабочую папку
Не давайте агенту доступ ко всему домашнему каталогу без необходимости.

### 2. Не храните секреты рядом
Уберите:
- API keys,
- credentials,
- production secrets,
- приватные токены
из рабочей директории, если они не нужны задаче.

### 3. Ограничивайте runtime-среду
Если возможно, запускайте такой workflow в sandboxed workspace.

### 4. Проверяйте, какие MCP включены
Browser automation и внешние интеграции — это особенно чувствительная зона.

### 5. Логируйте изменения
Хорошо, если все созданные и изменённые файлы фиксируются через Git.

## Что можно улучшить в этом файле

### 1. Уточнить имя файла
Сейчас есть несогласованность между:
- `setting.json`
- `settings.local.json`

Лучше выбрать один стандарт.

### 2. Добавить комментарий или документацию рядом
Сам JSON короткий, но без README его смысл неочевиден.

### 3. Разделить режимы
Полезно иметь хотя бы два профиля:
- `safe` — с ограниченными правами;
- `autonomous` — с полным набором разрешений.

### 4. Ограничить overly broad permissions
Если нужны только конкретные команды или только файловые действия в одной папке, лучше сужать доступ.
