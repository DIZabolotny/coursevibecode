## Тезаурус: подробное описание каждой возможности

### Ядро (общее для всех интерфейсов)

**Агентный цикл (анализ → действие → проверка).** Фундаментальный принцип работы Claude Code во всех интерфейсах: модель анализирует задачу, решает, какой инструмент вызвать (прочитать файл, выполнить команду, внести правку), выполняет действие, проверяет результат и переходит к следующему шагу. Этот цикл крутится автономно, пока задача не решена. Именно это отличает Claude Code от обычного чата: модель не просто отвечает текстом, а действует.

**Инструменты (Bash, Read, Write, Edit, Grep, Glob).** Набор встроенных инструментов, доступных модели. Bash — выполнение любых команд в терминале. Read — чтение файлов. Write — создание новых файлов. Edit — точечное редактирование существующих файлов (замена фрагмента, а не перезапись). Grep — поиск по содержимому файлов с поддержкой регулярных выражений. Glob — поиск файлов по паттерну имени (например, `**/*.py` найдёт все Python-файлы). Эти инструменты одинаковы во всех интерфейсах.

**MCP-серверы.** Внешние сервисы, подключённые через Model Context Protocol. Конфигурируются в файлах `.mcp.json` (уровень проекта) или `~/.claude.json` (глобально). Важный нюанс: MCP-серверы, настроенные в Claude Desktop (приложение для чата, а не Claude Code Desktop), живут в отдельном конфиге `claude_desktop_config.json` и не видны из Claude Code. Для Claude Code нужна своя конфигурация.

**Хуки (pre/post tool execution).** Пользовательский код, который автоматически выполняется до или после каждого вызова инструмента. Настраиваются в конфигурации и работают одинаково во всех интерфейсах. Примеры использования: логирование всех bash-команд в файл, блокировка команд с `rm -rf`, автоматическое форматирование кода после каждого Edit, отправка уведомления в Slack при завершении задачи.

**CLAUDE.md (настройки проекта).** Текстовый файл в корне проекта, который модель читает автоматически при запуске сессии в этой директории. Работает одинаково во всех интерфейсах. Может содержать описание архитектуры проекта, правила код-стайла, инструкции по сборке и тестированию, список запрещённых действий, контекст о бизнес-логике. Фактически это способ «обучить» модель вашему проекту один раз, а не объяснять заново в каждом диалоге.

**Субагенты.** Модель может порождать дочерние агенты, каждый из которых работает изолированно со своим набором инструментов и контекстом. Например, основной агент может запустить субагента для поиска по кодовой базе и одновременно другого для запуска тестов. Результаты возвращаются основному агенту. Это позволяет параллелизировать работу и не загрязнять основной контекст промежуточными данными.

**Slash-команды.** Встроенные и пользовательские команды, вызываемые через `/`. Примеры встроенных: `/help`, `/clear`, `/compact` (сжатие контекста), `/model` (смена модели). Пользовательские команды определяются в `.claude/commands` и могут запускать произвольные workflow. В VS Code Extension доступно подмножество команд — не все CLI-команды реализованы в графическом интерфейсе.

### Режимы работы

**Интерактивный диалог.** Стандартный режим: вы вводите запрос, модель отвечает и действует, вы видите результат и продолжаете диалог. Доступен везде, но интерфейс различается: в CLI — текстовый ввод в терминале, в VS Code — панель чата сбоку от редактора, в Desktop — полноценное графическое окно.

**Headless-режим (`claude -p`).** Запуск Claude Code без интерактивного интерфейса. Вы передаёте промпт как аргумент командной строки: `claude -p "найди все TODO в проекте и создай отчёт"`. Модель выполняет задачу и выводит результат в stdout. Это критическая функция для автоматизации: headless-режим позволяет встраивать Claude Code в bash-скрипты, cron-задачи, CI/CD-пайплайны, и любые другие процессы, которые запускаются без участия человека. Доступен только в CLI.

**Сторонние провайдеры (Bedrock, Vertex).** Возможность использовать Claude Code через Amazon Bedrock, Google Vertex AI или Azure Foundry вместо прямого API Anthropic. Актуально для корпоративных клиентов, которые обязаны маршрутизировать трафик через определённого облачного провайдера по соображениям безопасности, комплаенса или биллинга. Desktop App эту функцию не поддерживает — работает только через прямое API Anthropic.

**Полный bypass разрешений (`dontAsk`).** Режим, при котором модель выполняет все действия без запроса подтверждения: правка файлов, выполнение команд, удаление — всё автоматически. Опасный, но необходимый для полной автоматизации (например, в CI/CD-пайплайне). В CLI доступен по умолчанию. В Desktop — только через явное включение в настройках. В VS Code Extension — недоступен.

**Режим Plan (только чтение).** Модель анализирует задачу, читает файлы, строит план, но не вносит никаких изменений. Полезно, когда вы хотите сначала понять, что модель собирается делать, прежде чем дать ей разрешение. Доступен во всех интерфейсах.

**Автоодобрение правок (acceptEdits).** Промежуточный режим: модель автоматически вносит правки в файлы, но запрашивает подтверждение для выполнения bash-команд. Баланс между скоростью и безопасностью. Доступен везде.

### Контекст из IDE

**Автоконтекст: текущий открытый файл.** VS Code Extension автоматически передаёт модели информацию о том, какой файл у вас сейчас открыт в редакторе. Вам не нужно говорить «посмотри файл auth.py» — модель уже знает, что вы в нём работаете. В CLI и Desktop этой автоматики нет: нужно явно указать путь к файлу или позволить модели самой найти нужный файл через Glob/Grep.

**Автоконтекст: выделенный текст.** Если вы выделили фрагмент кода в VS Code и открыли панель Claude, модель автоматически увидит этот фрагмент и его контекст (имя файла, строки). Можно сразу спросить «что делает этот код?» или «перепиши с обработкой ошибок» без копипаста.

**Диагностика IDE (ошибки линтера, типов).** VS Code Extension автоматически передаёт модели ошибки и предупреждения из IDE: ошибки TypeScript, предупреждения ESLint, проблемы mypy. Модель видит их без дополнительных команд и может сразу предложить исправления. В CLI для аналогичного результата нужно явно запустить линтер через bash и дать модели прочитать вывод.

**Нативный diff-viewer (side-by-side).** Когда модель вносит правки в файлы, способ их отображения зависит от интерфейса. В CLI — текстовый diff в терминале (+ и - строки). В VS Code Extension — нативный side-by-side diff VS Code, тот же, что при `git diff`, с подсветкой изменений и возможностью принять/отклонить каждое изменение. В Desktop — собственный визуальный diff с возможностью комментирования.

**Горячая клавиша вставки ссылки на файл (Alt+K / Option+K).** Быстрый способ вставить в сообщение ссылку на файл с указанием диапазона строк: `@src/auth.py`. Модель получит точный контекст — не весь файл, а именно указанный фрагмент. Экономит токены и фокусирует модель на нужном участке кода.

**Поддержка Jupyter-ноутбуков.** В VS Code Extension модель может выполнять ячейки в открытых Jupyter-ноутбуках через специальный MCP-инструмент `mcp__ide__executeCode`. Это позволяет интерактивно работать с данными, запускать эксперименты, визуализировать результаты — всё внутри ноутбука. В CLI ноутбуки можно обрабатывать через bash (запуск `jupyter`, `papermill`), но без нативной интеграции с интерфейсом.

### Уникальные функции Desktop App

**Встроенный превью приложения (embedded browser).** Desktop App может запустить dev-сервер вашего приложения и показать результат прямо в окне — встроенный браузер с live-reload. Модель видит, как выглядит приложение, и может автоматически проверять визуальный результат своих правок. Ни CLI, ни VS Code Extension не имеют этой функции: в CLI нет графического вывода, в VS Code для этого нужно отдельно открывать браузер.

**Мониторинг PR + автофикс CI.** Desktop App может следить за открытыми pull request на GitHub. Если CI (continuous integration) упал — тесты не прошли, линтер нашёл ошибку — модель автоматически анализирует проблему, вносит исправления и пушит фикс. Может также автоматически мержить PR после прохождения всех проверок. Это фоновая автоматизация, которая работает, пока вы занимаетесь другими задачами.

**Параллельные сессии с git-изоляцией (worktrees).** В Desktop App каждая параллельная сессия может автоматически получить свой git worktree — изолированную копию репозитория. Это значит, что две сессии могут одновременно работать над разными задачами в одном проекте, не мешая друг другу: каждая в своей ветке, с собственной рабочей копией. В CLI вы можете создать worktrees вручную, но автоматической изоляции нет. В VS Code Extension — вкладки для параллельных сессий, но без git-изоляции.

**Scheduled Tasks (GUI-планировщик).** Графический интерфейс для создания и управления задачами по расписанию: ежедневные отчёты, еженедельные проверки, регулярные обновления. Каждая задача запускается в новой сессии, выполняет промпт и завершается. Управление через боковую панель: создание, редактирование, включение/выключение, просмотр истории запусков. В CLI встроенного планировщика нет — для аналогичной функциональности используется системный cron с headless-режимом.

**SSH-подключение к удалённым машинам.** Desktop App позволяет подключиться к удалённому серверу через SSH и запустить Claude Code там. Модель работает в файловой системе удалённой машины: читает логи сервера, правит конфигурации, перезапускает сервисы. Графический интерфейс при этом остаётся локальным — на вашем компьютере. В CLI вы тоже можете зайти на сервер по SSH и запустить `claude` оттуда, но Desktop предоставляет для этого удобный GUI с управлением сессиями.

**Облачное выполнение (Anthropic cloud).** Возможность запускать длительные задачи на облачной инфраструктуре Anthropic, а не на локальной машине. Полезно для ресурсоёмких операций: масштабный рефакторинг, обработка больших объёмов данных, параллельная работа с несколькими репозиториями. Ваш компьютер при этом свободен.

**Визуальное управление сессиями (sidebar).** Боковая панель со списком всех сессий: активных, завершённых, архивированных. Можно вернуться к предыдущей сессии, удалить ненужные, видеть статус каждой. В CLI управление сессиями ограничено: можно возобновить последнюю сессию через `claude --resume`, но визуального списка нет.

**Графический менеджер коннекторов.** Визуальный интерфейс для подключения внешних сервисов: Google Calendar, Slack, Linear, GitHub и других. Подключение через OAuth в несколько кликов, без редактирования JSON-файлов. В CLI для настройки MCP-серверов нужно вручную редактировать `.mcp.json` или `~/.claude.json`.

### Автоматизация и скриптование

**Встраивание в shell-скрипты.** Claude Code CLI можно вызывать из любого bash/zsh-скрипта как обычную команду. Например: скрипт, который клонирует репозиторий, запускает `claude -p "найди уязвимости"` и отправляет результат по email. Это превращает модель из интерактивного инструмента в компонент автоматизированного пайплайна.

**Пайпинг ввода/вывода (stdin/stdout).** Возможность передавать данные в Claude Code через pipe и получать результат на stdout. Пример: `cat error.log | claude -p "проанализируй ошибки"`. Или: `claude -p "сгенерируй SQL для создания таблицы users" > create_users.sql`. Стандартная unix-философия: программа как фильтр, который можно комбинировать с другими.

**Системный cron / launchd.** Создание задач на уровне операционной системы, которые выполняются независимо от любых приложений. Claude Code CLI может настроить crontab-запись (Linux) или launchd-сервис (macOS), который по расписанию запускает `claude -p "..."`. Компьютер включён — задача работает. Ни Desktop App, ни VS Code Extension не могут создавать системные задачи — у них есть только встроенный планировщик, привязанный к работающему приложению.

**GitHub Actions (CI/CD).** Запуск Claude Code в облаке GitHub при определённых событиях: новый pull request, push в ветку, по расписанию. Типичные сценарии: автоматическое ревью кода, генерация документации, исправление ошибок линтера, обновление changelog. Работает полностью в облаке, не требует вашего компьютера.

**Agent SDK (программный контроль).** Библиотека для Python и TypeScript, открывающая возможности Claude Code как программного API. Позволяет создавать собственных AI-агентов с кастомными инструментами, запускать их как сервисы, оркестрировать несколько агентов, обрабатывать очереди задач. Максимальный уровень контроля: вы определяете, какие инструменты доступны, как обрабатываются разрешения, когда агент останавливается, какая модель используется. Это инструмент для разработчиков, строящих продукты на базе Claude.

### Разработка

**Полный доступ к файловой системе ОС.** Во всех трёх интерфейсах Claude Code работает непосредственно в вашей операционной системе. Это принципиальное отличие от Cowork, где модель ограничена песочницей. Claude Code видит все ваши файлы, проекты, конфигурации. С этим связан и риск: модель может случайно повредить важные файлы, поэтому существуют режимы разрешений.

**Git commit / push / PR.** Полный цикл работы с git: модель может создавать осмысленные коммиты с описаниями, пушить изменения на удалённый репозиторий, создавать pull request через GitHub CLI с заголовком и описанием. Работает одинаково в CLI, VS Code и Desktop, хотя Desktop добавляет визуальный мониторинг PR.

**Запуск тестов и линтеров.** Модель может запустить тестовый фреймворк (`pytest`, `jest`, `go test`), прочитать результат, найти упавшие тесты, исправить код и перезапустить. Полный цикл red-green-refactor. Работает одинаково во всех интерфейсах, но VS Code Extension дополнительно получает диагностику IDE.

**Docker / контейнеры.** Управление Docker: сборка образов (`docker build`), запуск контейнеров (`docker run`), просмотр логов (`docker logs`), работа с docker-compose. Требует, чтобы Docker daemon был установлен и запущен на машине. Доступно во всех интерфейсах Claude Code.

**Tab-completion команд.** Автодополнение при вводе slash-команд и аргументов. Работает только в CLI (терминале), где есть стандартный механизм tab-completion. В графических интерфейсах (VS Code, Desktop) используется другой подход — выпадающие списки при вводе `/`.

**Быстрый bash (`!command`).** Ярлык в CLI: вместо того чтобы просить модель выполнить команду, вы можете ввести `!ls` или `!git status`, и команда выполнится напрямую в терминале, минуя модель. Удобно для быстрых проверок, не тратящих токены.

### Удобство и интерфейс

**Графический интерфейс.** VS Code Extension отображается как панель чата рядом с редактором кода. Desktop App — полноценное отдельное окно с sidebar, превью, менеджером сессий. CLI — текстовый интерфейс в терминале. Функциональность ядра одинакова, но способ взаимодействия различается: кто-то предпочитает текст, кто-то графику, кто-то хочет интеграцию с IDE.

**Множественные параллельные сессии.** Возможность вести несколько задач одновременно. В CLI — открываете несколько терминальных окон или вкладок. В VS Code Extension — вкладки внутри панели Claude. В Desktop App — сессии в sidebar с опциональной git-изоляцией через worktrees. Подход Desktop наиболее продвинутый: каждая сессия может автоматически получить собственную ветку и рабочую копию.

**Управление плагинами.** Установка и настройка плагинов. В CLI — через команды в терминале. В VS Code Extension и Desktop — через графический менеджер. Функциональность одинакова, различается только способ взаимодействия.

## Краткий вывод

Все три интерфейса используют один и тот же движок и набор инструментов. Выбор определяется стилем работы и конкретными задачами. CLI — для автоматизации, скриптования и максимального контроля. VS Code Extension — для ежедневной разработки в IDE с автоконтекстом из редактора. Desktop App — для визуальной работы, управления PR, планирования задач и работы с удалёнными серверами.
