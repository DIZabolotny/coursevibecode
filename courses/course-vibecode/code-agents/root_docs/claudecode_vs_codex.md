## Аннотация

Это документ для тех, кто хочет понять простыми словами, чем отличаются `Claude Code` и `Codex`, не погружаясь глубоко в технические детали.

Его цель:
- показать, что умеет каждый агент;
- объяснить, в чём они похожи, а в чём различаются;
- помочь выбрать, какой инструмент лучше подходит под конкретную задачу.

Практическая польза документа:
- он помогает понять, какой агент лучше подходит для работы с макетами, ТЗ и сборкой продукта;
- он показывает, какой агент удобнее для строгих настроек, ограничений, автоматизаций и повторяемой работы;
- он объясняет ключевые термины простым языком, чтобы документ был полезен не только разработчикам.

Проще говоря, это карта выбора между двумя кодовыми агентами: когда брать `Claude Code`, когда брать `Codex`, а когда полезно сочетать подходы обоих.

Это обновлённая версия сравнения. Теперь оно основано не на двух, а на трёх документах из этого репозитория:
- `education/root_docs/claudecode_process_1.md`
- `education/root_docs/claudecode-precess_2.md`
- `education/root_docs/codex_process_1.md`

Важно: второй документ про Claude Code сильно меняет картину. Раньше могло казаться, что Claude Code в этих материалах описан в основном как инструмент для длинного продуктового pipeline, а Codex — как более формальная и настраиваемая агентная система. После появления `education/root_docs/claudecode-precess_2.md` видно, что это уже не совсем так.

Теперь картина точнее такая:
- **Claude Code** в этих материалах показан и как инструмент для сборки продукта, и как серьёзная агентная система с настройками, permissions, hooks, sandbox, subagents и MCP.
- **Codex** в этих материалах по-прежнему очень силён как настраиваемая агентная платформа с `AGENTS.md`, `config.toml`, `approval_policy`, `sandbox_mode`, `rules`, `automations` и custom agents.

То есть сравнение стало более честным и более симметричным.

Этот документ написан простым языком для людей, которые используют AI-инструменты, но не обязаны быть разработчиками. Если встречается непонятный термин, в конце есть **тезаурус**.

## Короткий вывод

Если сказать совсем просто:

- **Claude Code** по этим материалам лучше показан как агент, который помогает пройти путь от идеи и макетов до рабочего продукта.
- **Codex** по этим материалам лучше показан как агент, которого особенно удобно жёстко настраивать, ограничивать, масштабировать на несколько сценариев и ставить на повторяемую работу.

Но после нового Claude Code документа разница уже не выглядит как:
- "Claude Code — просто про продукт";
- "Codex — просто про системное управление".

Теперь видно, что **оба агента являются полноценными агентными платформами**, просто сильные стороны у них чуть разные.

## Что изменилось после нового Claude Code документа

После `education/root_docs/claudecode-precess_2.md` пришлось скорректировать несколько важных выводов:

1. Раньше сравнение могло создавать впечатление, что у Claude Code skills и subagents больше похожи на пользовательский паттерн.  
Теперь видно, что у Claude Code это тоже официальный и подробно задокументированный слой.

2. Раньше могло казаться, что серьёзный контроль автономности, безопасности и разрешений сильнее раскрыт только у Codex.  
Теперь видно, что у Claude Code тоже есть очень мощная система:
- `permissions`;
- `permission modes`;
- `hooks`;
- `sandbox`;
- `settings` с приоритетами;
- project rules;
- headless;
- SDK;
- GitHub Actions.

3. Раньше Claude Code выглядел в основном как "оркестратор продуктового pipeline".  
Теперь видно, что он также хорошо раскрыт как **управляемая агентная оболочка**, а не только как методология работы от Figma до production.

Итог: **разрыв между Claude Code и Codex в части "системности" стал меньше**. Сравнение теперь уже не про то, есть ли у Claude Code control plane вообще. Он есть. Вопрос теперь скорее в том, **какой тип управления и какой стиль работы вам ближе**.

## Платформенное сравнение

| Аспект | Claude Code | Codex | Что это значит простыми словами |
| --- | --- | --- | --- |
| Как вообще понимать продукт | В новом документе Claude Code описан как агентная оболочка вокруг модели: с памятью, tools, policy layer, subagents, MCP и разными поверхностями запуска. | Codex описан как система из слоёв: модель, инструкции, tools, runtime controls и workflow extensions. | Оба продукта — это уже не "один умный чат". Это рабочие системы, которые можно настраивать по частям. Если агент ведёт себя не так, проблема обычно не только в prompt, а в одном из этих слоёв. |
| Что реально управляет поведением | Claude Code прямо делит управление на `soft guidance`, `hard control` и `execution mode`. | Codex делит управление на `instructions`, `tools`, `runtime controls` и `workflow extensions`. | И там и там prompt — только часть картины. Реальное поведение агента зависит ещё от файлов правил, режима запуска, ограничений, подключённых инструментов и разрешений. |
| Главные файлы и где они лежат | `CLAUDE.md`, `.claude/rules`, `.claude/settings.json`, `.claude/agents`, `.claude/skills`, `.mcp.json` и связанные user-level файлы. | `AGENTS.md`, `.codex/config.toml` или `~/.codex/config.toml`, `.codex/agents`, skills и rules. | В обоих случаях агентом лучше управлять через файлы проекта, а не через длинные случайные промпты в чате. Это значит, что настройки можно сделать один раз и потом использовать много раз. |
| Что важнее, если настройки противоречат друг другу | У Claude Code явно расписан порядок приоритетов: managed settings, CLI-аргументы, local settings, project settings, user settings. | У Codex тоже есть явный порядок: CLI, profile, project config, user config, system config, defaults. | Для обычного пользователя это важно так: если агент "не слушается", причина часто в том, что более сильная настройка перекрыла более слабую. И у Claude Code, и у Codex это можно разбирать системно, а не гадать. |
| Постоянные инструкции по проекту | У Claude Code это `CLAUDE.md`, дополнительные rules и память проекта. Есть иерархия, imports и path-scoped rules. | У Codex это `AGENTS.md`, который тоже собирается по слоям от общего к частному. | Оба инструмента умеют "помнить", как устроен проект и как с ним надо работать. Это удобно, потому что не нужно в каждом новом чате снова объяснять архитектуру, стиль и ограничения. |
| Skills | Во втором документе видно, что skills в Claude Code — это уже официальный слой, а не только самодельный паттерн. Они могут вызываться напрямую и подхватываться автоматически. | У Codex skills тоже официальный встроенный механизм, с metadata, progressive disclosure и управлением через config. | Оба агента умеют хранить повторяемые "мини-инструкции" для типовых задач. Простыми словами: если вы часто делаете одно и то же, можно не писать заново объяснение каждый раз. |
| Субагенты | У Claude Code subagents официальные: со своим контекстом, system prompt, tools, permissions, режимами работы в фоне и на переднем плане, а также отдельной историей работы. | У Codex subagents тоже официальные: есть built-in типы и custom agents в TOML. | Оба агента умеют делить большую задачу на нескольких помощников. Разница в ощущении такая: Claude Code сильнее раскрыт как система, где у каждого помощника могут быть свои разрешения и своя отдельная история работы, а Codex — как система с готовыми типами ролей и наглядной конфигурацией агентов. |
| Насколько гибко можно ограничить субагента | У Claude Code у subagent можно отдельно задавать tools, permission mode, max turns, model и skills. | У Codex у custom agent можно задавать model, reasoning effort, sandbox, MCP и skills config. | В обоих случаях можно делать "узких" помощников, которые умеют только то, что им нужно. Это полезно, когда не хочется давать каждому агенту полный доступ ко всему подряд. |
| Tools | У Claude Code есть built-in tools вроде `Read`, `Write`, `Edit`, `Bash`, `Glob`, `Grep`, `WebFetch`, `Agent`, а также MCP tools. | У Codex есть встроенные инструменты, включая shell, web search, apply patch, file tools, MCP и custom functions. | Оба агента могут не только отвечать текстом, но и делать реальные действия: читать, редактировать, искать, запускать команды, работать с вебом и звать внешние сервисы. |
| Permissions и approvals | У Claude Code permissions очень детальны: `allow`, `ask`, `deny`, `defaultMode`, path rules, дополнительные директории. | У Codex основа — `approval_policy`, `sandbox_mode`, network access и rules. | Оба умеют работать осторожно, но стиль разный. Claude Code лучше описан для точечной настройки "что именно можно делать этим инструментом". Codex лучше описан для выбора общего режима свободы агента и правил поверх него. |
| Режимы автономности | У Claude Code есть режимы вроде `default`, `acceptEdits`, `dontAsk`, `bypassPermissions`, `plan`. | У Codex есть `read-only`, `workspace-write`, `danger-full-access`, а также policies вроде `on-request`, `never`, `untrusted`, плюс готовые shorthand-режимы. | И там и там можно выбирать, насколько самостоятельным будет агент. Но в Claude Code акцент сильнее на том, как он обращается с конкретными действиями, а в Codex — на общих режимах свободы и ограничений. |
| Hooks и rules | У Claude Code есть hooks на события: до запуска tool, после, при запросе разрешения, при старте subagent и так далее. Hook может разрешать, запрещать, спрашивать, менять входные данные и добавлять контекст. | У Codex есть `rules`, которые хорошо подходят для контроля определённых команд и шаблонов поведения. | Здесь видно важное различие. Claude Code лучше подходит, если вы хотите очень умный и программируемый контроль "на лету". Codex лучше подходит, если вам нужен более простой и понятный набор правил без лишней сложности. |
| Sandbox | У Claude Code есть отдельный sandbox-блок с путями, доменами, сокетами и escape hatch для unsandboxed commands. | У Codex sandbox встроен в основные режимы и хорошо связан с network access и approval policy. | Оба умеют держать агента "на коротком поводке". Если говорить просто, sandbox — это безопасный коридор, за который агент не должен выходить без отдельного решения. |
| MCP | У Claude Code MCP — официальный способ подключать GitHub, Jira, Sentry, Figma, browser toolchains, внутренние API и другие системы. | У Codex MCP тоже официальный путь подключения внешних tools, docs, browser, GitHub, Figma, Sentry и так далее. | В этом пункте сходство очень большое. И Claude Code, и Codex можно расширять внешними сервисами, а не использовать только в пределах локальных файлов. |
| Работа без интерфейса | У Claude Code явно описаны REPL, headless mode, SDK и GitHub Actions. | У Codex явно описаны CLI, app, automations, worktrees и общий runtime-конфиг. | Если вы хотите запускать агента без ручного сидения в чате, оба подходят. Но Claude Code сильнее описан как инструмент для автоматического запуска из скриптов, своей программы и GitHub Actions, а Codex — как инструмент для работы через приложение и регулярных фоновых задач. |
| Browser/UI automation | У Claude Code есть browser-related operations через MCP, а для настоящего "кликнуть, ввести, пройти форму" отдельно объясняется `computer use`. | У Codex тоже есть browser MCP path и отдельный путь через OpenAI computer use. | Оба агента могут реально работать с интерфейсом, но только если им подключить правильный инструмент. Это важный практический момент: сам по себе prompt редко даёт магию "сам нажал кнопку". Нужен конкретный browser/tool layer. |
| Security best practices | У Claude Code явно проговариваются: read-only by default, review before approval, project-specific permissions, devcontainers/VM, аккуратность с third-party MCP и untrusted content. | У Codex отдельно разобраны risk-режимы, allowlists, sandbox, internet access, rules и опасность `--yolo`. | Оба продукта серьёзно относятся к безопасности. Но Claude Code сильнее раскрыт как система с детальной защитой на уровне hooks и permissions, а Codex — как система с ясными рабочими режимами и безопасными defaults. |
| Регулярные задачи по расписанию | В новых Claude Code материалах хорошо раскрыты headless, SDK и GitHub Actions, но не выделена такая же явная встроенная функция задач по расписанию внутри самого приложения, как у Codex. | У Codex есть отдельный официальный слой `automations`. | Если вам нужно, чтобы агент жил своей жизнью и сам что-то делал по расписанию внутри приложения, по этим материалам Codex выглядит удобнее. Если же вы готовы запускать через скрипты, Actions или свою обвязку, Claude Code тоже подходит. |
| Готовые переносимые роли для многих проектов | Claude Code это тоже умеет через subagents, skills, rules и settings, но больше как гибкая система, которую вы собираете сами. | Codex это тоже умеет и выглядит более "формально упакованным" через custom agents, config profiles и automations. | Простыми словами: оба подходят для системной работы на много проектов. Claude Code даёт больше ощущение гибкости и "собери под себя". Codex даёт больше ощущение готовой рамки, которую легче стандартизировать. |

## Сравнение по практическим сценариям

| Сценарий | Claude Code | Codex | Что это значит простыми словами |
| --- | --- | --- | --- |
| У меня есть макеты в Figma, и я хочу из них сделать понятное ТЗ | Это одна из самых сильных сторон `education/root_docs/claudecode_process_1.md`: анализ макетов, генерация ТЗ, проверка ТЗ против макетов, список вопросов. | В Codex-документе Figma есть, но как часть общего MCP-слоя, а не как подробно расписанный путь до ТЗ. | Если вы мыслите от дизайна, а не от кода, по этим материалам Claude Code выглядит заметно практичнее. Он лучше объяснён как инструмент, который превращает макеты в рабочее описание продукта. |
| Я хочу пройти длинный путь от идеи до рабочего продукта | У Claude Code подробно расписан путь `Figma -> ТЗ -> frontend + database -> SQLite -> Supabase -> deploy -> Playwright`. | У Codex такой длинный продуктовый pipeline не расписан настолько подробно. | Если вам нужна пошаговая дорожная карта "что делать после чего", Claude Code по этим материалам даёт более понятный маршрут. |
| Мне важно жёстко контролировать, что агенту можно и нельзя | Новый Claude Code документ даёт очень сильный набор: permissions, permission modes, hooks, sandbox. | Codex тоже очень силён: approval policy, sandbox, rules, internet access. | Здесь уже нельзя сказать, что сильная система контроля есть только у Codex. Теперь видно, что и у Claude Code она очень серьёзная. Разница скорее в стиле: Claude Code — более тонкая и программируемая настройка, Codex — более понятная система режимов и правил. |
| Мне нужны повторяемые навыки и заготовленные роли | Claude Code теперь тоже явно это поддерживает официально через skills и subagents. | Codex тоже поддерживает это официально и хорошо показывает custom agent setup. | Оба подходят. Если вам важнее гибкость и тонкое управление ролями — Claude Code выглядит очень сильно. Если важнее ясная, "собранная" рамка с явными профилями — Codex по этим материалам удобнее. |
| Мне надо запускать агента без ручной работы через терминал, CI или код | У Claude Code хорошо раскрыты headless mode, SDK и GitHub Actions. | У Codex хорошо раскрыты CLI, app, automations и общая конфигурация среды. | Если вы хотите встроить агента в свои процессы как движок или часть automation stack, Claude Code по этим материалам особенно хорошо раскрыт. Если вы хотите удобную пользовательскую среду с recurring tasks внутри самого продукта, Codex выглядит сильнее. |
| Мне нужны регулярные фоновые задачи | У Claude Code это можно строить, но по этим материалам это больше через headless/Actions/свои сценарии. | У Codex это отдельная официальная возможность. | Если для вас важно "поставил на расписание и забыл", у Codex здесь есть более прямой путь. |
| Мне нужно реально кликать сайт, заполнять формы, ходить по UI | У Claude Code для этого отдельно объяснён `computer use` и browser-oriented tooling. | У Codex для этого тоже есть browser MCP path и общий OpenAI path через computer use. | Оба агента подходят. Главное — понимать, что для настоящей UI-автоматизации нужен специальный инструмент, а не только хороший prompt. |
| Я хочу использовать оба инструмента вместе, а не выбирать один навсегда | Claude Code в такой связке удобен как агент, который берёт уже подготовленный план и долго тянет исполнение. | Codex в такой связке удобен как агент, который помогает спорить с ТЗ, искать дыры и потом делать ревью результата. | Если вам не хочется выбирать один инструмент "навсегда", разумная рабочая схема может выглядеть так: `Codex -> Claude Code -> Codex`. Сначала усиливаете постановку, потом исполняете, потом снова ревьюите. |
| Мне нужно обучать неразработчиков работе с агентами | Claude Code выигрывает за счёт первого документа, где очень подробно разложен сам продуктовый pipeline. | Codex выигрывает в системном объяснении настройки и управления агентом. | Если ваша цель — научить человека строить продукт шаг за шагом, Claude Code понятнее. Если цель — научить человека дисциплинированно настраивать самого агента и его режимы, Codex удобнее. |
| Я хочу один агентный стек для многих проектов | У Claude Code теперь для этого есть уже вполне серьёзная основа: settings, rules, skills, subagents, hooks, MCP. | У Codex это тоже одна из сильных сторон: `AGENTS.md`, `config.toml`, agents, rules, automations. | И там и там это возможно. Выбор здесь скорее зависит от предпочтения: более гибкая и программируемая система против более формально упакованной системы. |
| Я хочу, чтобы агент не только понимал проект, но и реально собирал продукт | Claude Code по этим материалам сильнее, потому что продуктовый слой и длинный путь сборки описаны глубже. | Codex по этим материалам сильнее как управляемая агентная платформа, но не как подробно расписанный продуктовый pipeline. | Если вы хотите "дайте мне агента, который поможет провести проект через все стадии", Claude Code выглядит первым кандидатом. |

## Ещё один полезный угол зрения

Новый дополнительный документ дал ещё одну полезную мысль: **эти инструменты не очень полезно сравнивать только по списку функций**.

Почему:
- оба умеют читать проект;
- оба умеют менять код;
- оба умеют писать тесты;
- оба умеют делать ревью;
- оба умеют работать по инструкциям;
- оба умеют работать достаточно автономно.

Из-за этого сухое сравнение вида "этот умеет X, а этот умеет Y" часто мало что даёт.

Более полезный вопрос такой:

**Как агент ведёт себя в длинной работе, когда у него уже есть хорошее ТЗ, правила и доступы?**

И вот здесь появляется более живая, практическая разница.

### Рабочая гипотеза из практики

По новому альтернативному документу полезно держать в голове такую рабочую гипотезу:

- **Codex** часто полезнее как критик, редактор постановки, усилитель ТЗ и ревьюер.
- **Claude Code** часто удобнее как длинный исполнитель, который идёт по уже подготовленному плану с меньшим трением.

Это не означает:
- что Codex не умеет долго исполнять;
- что Claude Code не умеет думать;
- что их обязательно нужно использовать только вместе.

Это означает только одно:

**в части практик людям может быть удобно по-разному распределять роли между двумя зрелыми и уже довольно близкими по возможностям инструментами.**

### Важная оговорка

Этот вывод нельзя считать вечной истиной.

Часть разницы может идти не из "ума" агента как такового, а из:
- текущих настроек среды;
- режима approvals;
- sandbox;
- project rules;
- hooks;
- того, насколько хорошо подготовлены ТЗ и ограничения.

То есть правильнее читать эту разницу так:

- это **полезная рабочая гипотеза из практики**;
- но не **абсолютный закон природы**.

### Если использовать оба вместе

Тогда практичная схема может выглядеть так:

1. `Codex` помогает спорить с постановкой, ужесточать ТЗ и искать слабые места.
2. `Claude Code` берёт уже подготовленный пакет и тащит длинное исполнение.
3. `Codex` снова включается как ревьюер и критик результата.

В коротком виде:

`Codex -> Claude Code -> Codex`

Такая связка не обязательна, но для некоторых рабочих стилей она действительно может быть полезной.

## Где Claude Code теперь выглядит особенно сильным

После появления второго документа у Claude Code хорошо видно сразу две большие сильные зоны.

### 1. Сильный продуктовый слой

Это осталось из первого большого документа:
- Figma;
- user flow;
- ТЗ;
- data contract;
- frontend;
- database;
- SQLite;
- Supabase;
- deploy;
- Playwright.

То есть Claude Code по этим материалам очень хорошо раскрыт как агент, который помогает не просто "что-то сделать", а провести продукт через длинную цепочку этапов.

### 2. Сильный слой контроля и политики

Это уже очень усилил второй документ:
- permissions;
- hooks;
- sandbox;
- settings precedence;
- subagents;
- skills;
- headless;
- SDK;
- GitHub Actions;
- security practices.

Поэтому теперь Claude Code нельзя честно описывать только как "агента для продуктового pipeline". Он также выглядит как серьёзная платформа для управляемой автономной работы.

### 3. Особенно интересен для людей, которые хотят программируемый контроль

Если вам важно не просто "можно/нельзя", а:
- что делать при конкретном tool call;
- как логировать действия;
- как подменять входные данные;
- как запускать действия на событиях,

то hooks в Claude Code делают его особенно интересным. В этих материалах это один из самых сильных моментов.

## Где Codex по-прежнему выглядит особенно сильным

После обновления сравнения сильные стороны Codex не исчезли. Просто теперь они выглядят более конкретно.

### 1. Очень ясная модель управления режимами работы

У Codex по этим материалам по-прежнему очень хорошо видны:
- `AGENTS.md`;
- `config.toml`;
- `approval_policy`;
- `sandbox_mode`;
- `rules`;
- `web_search`;
- `internet access`;
- `automations`.

Это создаёт очень понятную модель: есть инструкции, есть глобальные режимы, есть отдельные правила, есть автоматизации.

### 2. Удобный "собранный" образ платформы

Codex выглядит как система, где из коробки проще мыслить такими объектами:
- профили;
- custom agents;
- built-in agent types;
- recurring automations;
- worktrees;
- config-driven setup.

Для многих людей это удобно именно психологически. Не потому, что другой продукт этого не умеет, а потому, что тут всё очень ясно разложено как набор крупных блоков.

### 3. Особенно силён там, где нужна повторяемая рабочая схема

Если вы хотите не один раз решить задачу, а выстроить постоянную рабочую среду:
- для нескольких проектов;
- для регулярных фоновых задач;
- для контролируемого shell/web режима;
- для повторяющихся проверок,

то Codex по этим материалам остаётся очень сильным кандидатом.

## Простые правила выбора

### Выбирайте Claude Code, если

- у вас всё начинается с макетов, экранов и логики продукта;
- вы хотите превратить дизайн в понятный план работ;
- вам нужен длинный путь от идеи до рабочего результата;
- вам важны Figma, ТЗ, browser testing, deploy и проверка продукта целиком;
- вам нужен более гибкий и "программируемый" контроль над действиями агента;
- вы хотите встраивать агента в headless-сценарии, SDK или GitHub Actions.

### Выбирайте Codex, если

- вам нужна понятная и строгая система режимов свободы агента;
- вы хотите отдельно управлять инструкциями, конфигами, агентами и автоматизациями;
- вам важны встроенные recurring automations;
- вы хотите устойчивую рабочую схему на много проектов;
- вам ближе конфигурационный подход: профили, custom agents, policies, rules, sandbox modes.

### Выбирайте смешанный подход мышления, если

- вы хотите у Claude Code взять продуктовый и workflow-подход;
- а у Codex взять ясную дисциплину по режимам, профилям, безопасным настройкам и регулярным автоматизациям.

Это очень сильная комбинация:
- от Claude Code взять способ вести проект;
- от Codex взять способ дисциплинированно настраивать среду и повторяемые режимы работы.

## Самое важное различие одной фразой

По этим материалам **Claude Code** лучше показан как агент, который помогает **вести продукт от идеи до запуска и при этом даёт очень гибкий контроль над поведением**, а **Codex** лучше показан как агент, которого особенно удобно **строить как стабильную, конфигурируемую и повторяемую рабочую платформу**.

## Тезаурус

Ниже объяснения всех терминов максимально простым языком.

**`CLAUDE.md`**  
Файл с постоянными инструкциями для Claude Code. Простыми словами: место, где вы заранее объясняете агенту, как работать с этим проектом.

**`AGENTS.md`**  
Похожий по смыслу файл у Codex. Это тоже слой постоянных правил, который задаёт стиль работы и ограничения по проекту.

**`settings.json`**  
Файл настроек Claude Code. Через него задают permissions, sandbox и другие важные ограничения и разрешения.

**`config.toml`**  
Файл настроек Codex. Через него задают модель, режимы доступа, MCP, skills и другие системные параметры.

**Skill**  
Повторяемая инструкция для типовой задачи. Например: "как делать ревью", "как проверять релиз", "как работать с макетами". Это как маленькая методичка.

**Subagent / субагент**  
Отдельный помощник внутри большой задачи. Главный агент может отправить ему часть работы и потом получить обратно результат.

**Custom agent**  
Заранее настроенный собственный тип агента или помощника. Например: отдельный ревьюер, исследователь или агент для релизной проверки.

**Tool**  
Конкретный инструмент, которым агент пользуется: чтение файлов, запись, bash-команды, поиск, web fetch, внешние сервисы и так далее.

**Permission**  
Разрешение на действие. Например: можно ли агенту читать файл, запускать команду или обращаться к сети.

**Approval**  
Запрос подтверждения. Если агент доходит до чувствительного действия, он должен спросить у человека: "можно ли продолжать?"

**Permission mode**  
Общий режим поведения агента по отношению к разрешениям. Например: всё ли спрашивать, часть действий одобрять автоматически или почти ничего не делать без явного allow.

**Sandbox**  
Безопасная ограниченная среда. Простыми словами: агент работает в загоне, за пределы которого не должен выходить без отдельного разрешения.

**Hook**  
Дополнительное правило, которое срабатывает на событии. Например: перед запуском команды, после записи файла или при старте субагента. Это как автоматический "контролёр" или "надзиратель".

**Rule / rules**  
Правила, которые ограничивают или разрешают определённые действия. Обычно они проще hooks и лучше подходят для понятных фиксированных ограничений.

**MCP**  
Способ подключить к агенту внешние сервисы. Например Figma, GitHub, браузер, базу данных, Jira или внутренний API.

**Headless mode**  
Запуск агента без обычного интерактивного чата. Полезен для скриптов, автоматизаций и серверных сценариев.

**REPL**  
Обычный интерактивный режим в терминале, где вы общаетесь с агентом шаг за шагом вживую.

**SDK**  
Набор средств для программистов, чтобы встроить агента в свой код или систему. Простыми словами: способ использовать агента как часть собственной программы.

**GitHub Actions / CI**  
Система автоматических действий в репозитории. Например: запустить проверку, сборку или агента после коммита.

**Profile / профиль**  
Готовый набор настроек под определённый режим работы. Например: безопасный режим, режим разработки или почти полностью автономный режим.

**Figma**  
Сервис для макетов интерфейсов. Там обычно рисуют экраны, кнопки, формы и переходы между экранами.

**ТЗ**  
Техническое задание. Простыми словами: письменный понятный план того, что именно должен уметь будущий продукт.

**User flow**  
Путь пользователя через продукт. Например: открыл экран, нажал кнопку, попал на форму, сохранил данные, увидел результат.

**Data contract**  
Согласованное описание данных. Простыми словами: единый договор о том, как называются поля и что куда передаётся, чтобы интерфейс и база понимали друг друга одинаково.

**SQLite**  
Очень простая локальная база данных, часто в одном файле. Удобна для ранней проверки логики без сложной инфраструктуры.

**Supabase**  
Готовая облачная платформа для базы данных, авторизации и серверной части приложения.

**Deploy / деплой**  
Перевод проекта из локальной разработки в реальное рабочее окружение, где им уже можно пользоваться.

**Playwright**  
Инструмент для автоматической проверки сайта через браузер. Может открывать страницы, кликать, вводить текст и смотреть, всё ли работает.

**Web search**  
Поиск в интернете, который агент может использовать для получения внешней информации.

**Computer use**  
Специальный инструмент, который позволяет агенту выполнять реальные действия в интерфейсе: двигать мышь, нажимать, вводить текст, делать скриншоты.

**Automation**  
Автоматическая задача, которая запускается сама: по расписанию, по событию или в фоне.

**Transcript / история работы**  
Сохранённая история того, что делал агент или субагент во время своей работы.

**Worktree**  
Отдельная рабочая копия проекта. Простыми словами: отдельный стол для параллельной работы, чтобы не мешать основной.

**Policy layer**  
Слой правил и ограничений, который решает, что агенту разрешено, а что нет.

**Runtime**  
Среда, в которой агент прямо сейчас работает. От неё зависит, какие инструменты доступны, есть ли сеть, какие включены ограничения и что агент вообще может делать.

## Итог

После нового Claude Code документа итог уже такой:

- **Claude Code** — это не просто инструмент для продуктового pipeline. Это и полноценная агентная система с subagents, skills, permissions, hooks, sandbox, MCP, headless и security layer.
- **Codex** — это не просто набор режимов и конфигов. Это тоже полноценная агентная система с сильным control plane, custom agents, skills, rules, MCP, web access и automations.

Главное различие теперь лучше понимать не как  
**"у одного есть серьёзное управление, а у другого нет"**,  
а как  
**"оба серьёзные, но один сильнее в продуктовой и программируемой гибкости, а другой — в конфигурируемой, повторяемой и более формально упакованной рабочей модели"**.

Если задача звучит так:

**"помоги мне пройти путь от макетов до рабочего продукта и при этом дай очень гибкий контроль над поведением агента"**,  
то по этим материалам первым кандидатом выглядит **Claude Code**.

Если задача звучит так:

**"дай мне агента, которого удобно настраивать через конфиги, правила, режимы, профили и регулярные automation-сценарии"**,  
то по этим материалам первым кандидатом выглядит **Codex**.
