# Codex: как устроены `AGENTS.md`, `.codex/config.toml`, `.rules` и `requirements.toml`

Этот документ объясняет простыми словами, как в Codex устроены уровни памяти, инструкций, настроек и ограничений.

Здесь разберем:

- что делает `AGENTS.md`;
- что делает `.codex/config.toml`;
- зачем нужны `.rules`;
- где появляется `requirements.toml`;
- как все это не перепутать.

Документ написан для людей без инженерного бэкграунда.

Проверено по официальной документации OpenAI по Codex на 20 марта 2026 года.

## Короткий ответ за 1 минуту

У Codex есть не один, а сразу несколько слоев управления.

1. `AGENTS.md`
Это слой постоянных инструкций и контекста.
Он говорит агенту:
- что это за проект;
- как здесь принято работать;
- что важно;
- какой результат считается хорошим.

2. `.codex/config.toml`
Это слой реальной конфигурации.
Он задает:
- модель;
- sandbox;
- approval policy;
- MCP;
- профили;
- другие рабочие параметры клиента.

3. `.rules`
Это не "текстовые правила для стиля".
В Codex `.rules` нужны для другого:
- разрешать;
- запрашивать подтверждение;
- запрещать
конкретные команды вне sandbox.

4. `requirements.toml` и `managed_config.toml`
Это уровень организации или администраторов.
Он нужен там, где компания хочет не просто советовать, а реально ограничивать или навязывать стартовые значения.

Если совсем коротко:

- хотите объяснить агенту, как думать и работать в проекте: `AGENTS.md`;
- хотите задать рабочие настройки клиента: `.codex/config.toml`;
- хотите управлять исключениями для команд за пределами sandbox: `.rules`;
- хотите жесткие организационные ограничения: `requirements.toml`.

## Самая важная мысль

В Codex очень легко смешать четыре разные вещи:

- инструкции;
- настройки;
- исключения для команд;
- организационные запреты.

Но это разные механизмы.

`AGENTS.md` не равен `config.toml`.
`.rules` не равны `AGENTS.md`.
`requirements.toml` не равен обычным пользовательским настройкам.

Самая полезная бытовая формула такая:

- `AGENTS.md` отвечает на вопрос "как здесь принято работать";
- `config.toml` отвечает на вопрос "в каком режиме стартует Codex";
- `.rules` отвечает на вопрос "что делать с командами вне sandbox";
- `requirements.toml` отвечает на вопрос "что в этой организации вообще нельзя нарушить".

## Очень важное отличие от Claude Code

Если вы раньше читали про Claude Code, не переносите логику механически.

В Claude Code слово `rules` обычно означает текстовые правила для поведения модели.

В Codex `.rules` означает другое:
это правила выполнения команд вне sandbox через `prefix_rule()`.

То есть:

- в Codex текстовые инструкции живут в `AGENTS.md`;
- а `.rules` в Codex управляют эскалацией команд и доступами.

Это одна из самых важных вещей во всей теме.

## Общая карта уровней

Удобно держать в голове вот такую схему.

```text
Enterprise / organization level
├── cloud-managed requirements
├── /etc/codex/requirements.toml
└── /etc/codex/managed_config.toml

User level
~/.codex/
├── AGENTS.override.md or AGENTS.md
├── config.toml
└── rules/
    └── default.rules

Project level
project-root/
├── AGENTS.override.md or AGENTS.md
├── .codex/
│   ├── config.toml
│   └── rules/
│       └── project.rules
└── subdir/
    ├── AGENTS.override.md or AGENTS.md
    └── .codex/
        └── config.toml
```

Важно:

- `AGENTS.md` может быть глобальным и вложенным;
- `.codex/config.toml` тоже может быть глобальным и проектным;
- `.rules` тоже могут быть на пользовательском и командном уровне;
- организационный слой стоит выше всего и может ограничивать локальные решения.

## 1. Как устроен `AGENTS.md`

### Что это такое

`AGENTS.md` в Codex это постоянные инструкции для агента.

По официальной документации OpenAI Codex читает `AGENTS.md` до начала работы.

Простыми словами:
это не программа, а памятка.

Она объясняет агенту:

- что это за проект;
- кто аудитория;
- какие правила важны;
- как выглядит хороший результат;
- что здесь нельзя ломать.

### Где он может лежать

По документации Codex ищет инструкции в нескольких местах.

1. Глобальный уровень пользователя

В домашней папке Codex:

- `~/.codex/AGENTS.override.md`
- если его нет, тогда `~/.codex/AGENTS.md`

На этом уровне используется только первый непустой файл.

2. Проектный уровень

Дальше Codex идет от корня проекта к вашей текущей рабочей папке.

В каждой папке по пути он проверяет:

- `AGENTS.override.md`
- `AGENTS.md`
- fallback-имена из `project_doc_fallback_filenames`, если они настроены

Из каждой папки берется максимум один файл.

### Какой порядок приоритета

OpenAI описывает логику так:

- Codex собирает цепочку инструкций от корня вниз;
- более близкие к текущей папке файлы добавляются позже;
- поэтому именно они уточняют и перекрывают более общие инструкции.

Если сказать по-человечески:

- инструкция в домашней папке задает ваши общие привычки;
- инструкция в корне проекта задает правила проекта;
- инструкция в конкретной подпапке уточняет работу именно в этой зоне.

### Бытовой пример

Представьте три таблички.

Первая табличка висит у вас дома:
- "Работай спокойно, не делай лишнего, сначала покажи план".

Вторая висит на двери проекта:
- "Этот проект для нетехнической аудитории. Пиши простым языком".

Третья висит на двери папки `docs`:
- "В этой папке каждое действие в интерфейсе объясняй буквально".

Когда агент работает именно в `docs`, третья табличка важнее двух предыдущих.

### Что полезно хранить в `AGENTS.md`

Хорошо подходят:

- цель проекта;
- аудитория;
- стиль изложения;
- правила редактирования;
- структура файлов;
- определение слова "готово";
- запретные зоны;
- нужные команды проверки.

### Что не стоит хранить в `AGENTS.md`

Плохо подходят:

- секреты;
- токены;
- пароли;
- длинные журналы истории;
- старые правила, которые уже не действуют;
- все подряд без структуры.

В документации OpenAI прямо есть мысль, что слишком длинные инструкции надо дробить.

### Важный технический нюанс

Codex ограничивает общий размер цепочки проектных инструкций.
По документации лимит по умолчанию составляет `32 KiB` через `project_doc_max_bytes`.

Практический вывод простой:

- не пишите многотомник в одном файле;
- лучше держать короткие и практичные инструкции;
- если проект большой, раскладывайте инструкции по вложенным папкам.

## 2. Как устроен `.codex/config.toml`

### Что это такое

`.codex/config.toml` это уже не памятка, а настоящая конфигурация клиента.

Через него управляют такими вещами, как:

- модель;
- provider;
- sandbox;
- approval policy;
- writable roots;
- MCP;
- профили;
- часть поведения среды.

Если `AGENTS.md` отвечает за "договоренности",
то `config.toml` отвечает за "режим запуска".

### Где лежит конфигурация

По документации OpenAI основные уровни такие:

- `~/.codex/config.toml` — ваши личные дефолты;
- `.codex/config.toml` внутри проекта — проектные overrides;
- системный файл тоже возможен, например `/etc/codex/config.toml` на Unix.

### Какой порядок приоритета

Документация Codex дает явный порядок приоритета.
Сверху вниз:

1. CLI-флаги и `--config`
2. значения профиля из `--profile <name>`
3. проектные `.codex/config.toml` от корня проекта вниз до текущей папки
4. пользовательский `~/.codex/config.toml`
5. системный `/etc/codex/config.toml`
6. встроенные значения по умолчанию

Ключевая мысль:

- ближний проектный файл важнее дальнего;
- временный запуск через CLI важнее обычного файла;
- профиль сидит выше обычных файлов, но ниже прямых CLI-override.

### Как работают подпапки

Codex может читать не только один `.codex/config.toml`, а несколько.

По документации:

- он идет от корня проекта к текущей папке;
- подгружает каждый `.codex/config.toml` по пути;
- если один и тот же ключ задан несколько раз, выигрывает ближайший к текущей папке файл.

Это значит, что в большом репозитории можно делать:

- общие проектные defaults в корне;
- более локальные настройки в конкретной подпапке.

### Когда проектные настройки вообще загружаются

Не всегда.

OpenAI прямо пишет:
проектные `.codex/config.toml` загружаются только для trusted project.

Если проект помечен как недоверенный, Codex игнорирует проектные `.codex`-слои и падает назад на:

- пользовательский уровень;
- системный уровень;
- встроенные defaults.

Это важный защитный механизм.

### Что имеет смысл держать в user config

Хорошие кандидаты:

- ваша любимая модель по умолчанию;
- ваш общий стиль approval policy;
- ваши MCP-серверы;
- ваши привычные персональные defaults.

### Что имеет смысл держать в project config

Хорошие кандидаты:

- безопасный режим для команды;
- нужные проекту sandbox и writable roots;
- проектные MCP;
- project-specific модель и provider;
- локальные настройки для этой кодовой базы.

### Что не стоит делать

Плохая идея:

- пытаться запихнуть в `config.toml` смысл проекта и редакционные правила;
- хранить в одном месте и личные вкусы, и организационную политику, и локальные исключения;
- раздувать конфиг без явной причины.

Если вопрос звучит как:
"Как агент должен писать и думать?"

это почти всегда `AGENTS.md`, а не `config.toml`.

## 3. Как устроены `.rules`

### Что это такое

В Codex `.rules` не описывают тон речи и не заменяют `AGENTS.md`.

Они нужны для одного конкретного класса задач:
контролировать, какие команды Codex может запускать вне sandbox.

Правило задается через `prefix_rule()`.

Оно может:

- разрешить команду;
- требовать подтверждение;
- запретить команду.

### Где живут `.rules`

По документации rules лежат в `rules` внутри Team Config locations.

На практике для пользователя важнее всего знать два места:

- `~/.codex/rules`
- `.codex/rules` в проекте

Когда вы в интерфейсе добавляете команду в allow list, Codex пишет это в пользовательский слой:

`~/.codex/rules/default.rules`

### Какой у rules смысл на бытовом языке

Представьте шлагбаум.

Не все команды одинаково опасны.

Например:

- одну команду можно пропускать автоматически;
- другую лучше пропускать только после вопроса;
- третью вообще нельзя пускать.

`.rules` и есть такой шлагбаум.

### Три решения у rules

1. `allow`
Команду можно запускать без лишнего вопроса вне sandbox.

2. `prompt`
Перед запуском нужно спрашивать.

3. `forbidden`
Команда запрещена.

Если совпало несколько правил, Codex применяет более жесткое решение.
Порядок такой:

`forbidden` > `prompt` > `allow`

### Хороший бытовой пример

Если вы хотите, чтобы Codex не молча выходил наружу с `gh pr view`, можно сделать правило с `prompt`.

Если хотите полностью запретить разрушительный класс команд, можно сделать `forbidden`.

### Очень важная мысль про shell wrappers

OpenAI отдельно предупреждает:
когда команда выглядит как `bash -lc "..."`, внутри может быть целая цепочка действий.

Если скрипт простой, Codex может разложить его на отдельные команды и применить правила к каждой.

Но если там есть:

- переменные среды;
- подстановки;
- редиректы;
- wildcard;
- сложный flow,

тогда вся строка рассматривается как одна большая команда.

Практический вывод:

- не стройте безопасность на надежде, что длинная shell-строка "как-нибудь сама" разберется;
- предпочитайте ясные, устойчивые префиксы;
- не включайте секреты inline в такие команды.

### Как проверять `.rules`

По документации для проверки можно использовать:

```text
codex execpolicy check
```

Это полезно, когда вы не уверены, как именно сработает правило.

## 4. Как связаны `sandbox_mode` и `approval_policy`

Это главный практический участок для большинства людей.

### `sandbox_mode`

Это ответ на вопрос:
"Что агенту физически разрешено делать?"

По документации основные режимы такие:

- `read-only`
- `workspace-write`
- `danger-full-access`

### `approval_policy`

Это ответ на вопрос:
"В какие моменты агент должен остановиться и спросить?"

Основные варианты:

- `untrusted`
- `on-request`
- `never`

### Простой перевод на человеческий язык

`sandbox_mode` говорит:
- где проходит физическая граница.

`approval_policy` говорит:
- где проходит граница согласования.

То есть это две разные оси.

### Что важно помнить о пресетах

По документации OpenAI:

- `--full-auto` это не полный беспредел;
- это связка `workspace-write` + `on-request`.

А вот реально опасный режим это:

- `danger-full-access` + `never`

или шорткат вроде `--yolo`.

### Полезный практический вывод

Новичку почти всегда подходит такая лестница:

1. сначала `read-only`;
2. потом `workspace-write`;
3. и только потом, если есть реальная причина, более широкий режим.

## 5. Дополнительная защита самой папки `.codex`

Это важный нюанс, который легко пропустить.

По документации OpenAI в стандартном `workspace-write` режиме сама папка `.codex` защищена как read-only внутри writable root.

То же относится и к:

- `.git`
- `.agents`

Практический смысл:

- Codex по умолчанию не должен свободно переписывать собственные проектные настройки в `.codex`;
- это снижает риск случайной самоперенастройки проекта;
- но это не отменяет необходимости думать, какой режим вы вообще включили.

## 6. Где начинается организационный уровень

До этого мы говорили про пользовательские и проектные уровни.

Но в Codex есть еще слой организации.

И тут появляются два разных файла:

### `requirements.toml`

Это admin-enforced constraints.

Простыми словами:

- пользователь не может их обойти обычным локальным конфигом;
- если локальный конфиг конфликтует с enforced-правилом, Codex откатывается к совместимому значению и уведомляет пользователя.

По документации через `requirements.toml` можно ограничивать:

- approval policy;
- sandbox mode;
- web search mode;
- MCP allowlist;
- некоторые feature flags;
- restrictive command rules.

### `managed_config.toml`

Это не жесткий запрет, а managed defaults.

Простыми словами:

- организация задает стартовые значения;
- пользователь может поменять их внутри текущей сессии;
- но при следующем запуске Codex снова вернется к managed defaults.

### Почему это важно

Это две разные идеи:

- `requirements.toml` говорит "ниже этого барьера нельзя";
- `managed_config.toml` говорит "стартуем вот с этого значения, даже если пользователь любит по-другому".

## 7. Полная картина приоритетов

Если собрать все в одну практическую схему, получится так.

### Для инструкций

`AGENTS.md`-цепочка:

1. `~/.codex/AGENTS.override.md` или `~/.codex/AGENTS.md`
2. `AGENTS.override.md` / `AGENTS.md` от корня проекта до текущей папки
3. чем ближе к текущей папке, тем инструкция важнее

### Для обычной конфигурации

`config.toml`-цепочка:

1. CLI flags и `--config`
2. профиль
3. проектные `.codex/config.toml` по пути от корня вниз
4. `~/.codex/config.toml`
5. системный `config.toml`
6. встроенные defaults

### Для командных исключений

`.rules`-цепочка:

- user и project rules объединяются;
- для совпадающих правил побеждает более строгий результат;
- allow list из интерфейса по умолчанию пишет в `~/.codex/rules/default.rules`

### Для организации

`requirements.toml` стоит выше локальной самодеятельности.

`managed_config.toml` задает стартовые значения поверх локального конфига.

## 8. Какую архитектуру лучше выбрать обычному пользователю

Если нужен простой и устойчивый вариант, схема обычно такая.

### Личный слой

В `~/.codex/AGENTS.md` держите:

- стиль общения;
- общие рабочие привычки;
- ваши личные предпочтения.

В `~/.codex/config.toml` держите:

- любимую модель;
- базовый sandbox;
- ваши глобальные MCP и привычные defaults.

### Слой проекта

В `project-root/AGENTS.md` держите:

- смысл проекта;
- аудиторию;
- правила работы;
- definition of done.

В `project-root/.codex/config.toml` держите:

- безопасные project defaults;
- MCP проекта;
- project-specific runtime settings.

### Слой подпапки

Если у конкретной зоны проекта отдельная логика, добавляйте:

- `subdir/AGENTS.md`
- при реальной необходимости `subdir/.codex/config.toml`

### Слой исключений

Если нужен отдельный пропуск для внешней команды, используйте:

- `.codex/rules/*.rules`

Но не пытайтесь этим заменить `AGENTS.md`.

### Слой организации

Если задача звучит как:

- "всем в компании нельзя full access";
- "можно только определенные MCP";
- "всем ставим безопасные стартовые значения";

это уже история не про обычный проектный файл, а про:

- `requirements.toml`
- `managed_config.toml`

## 9. Практический playbook по частым кейсам

Ниже не теория, а бытовые ситуации.

### Кейс 1. Я хочу, чтобы Codex во всех проектах говорил и работал в моем стиле

Что делать:

1. Откройте домашнюю папку Codex.
2. Найдите или создайте `~/.codex/AGENTS.md`.
3. Запишите туда только ваши общие привычки.

Что туда хорошо подходит:

- отвечай кратко;
- сначала давай план на больших задачах;
- после правок запускай проверку;
- не используй лишний пафос.

Что туда не надо класть:

- правила конкретного проекта;
- секреты;
- историю одной отдельной задачи.

### Кейс 2. Я хочу, чтобы один конкретный репозиторий запускался в более безопасном режиме

Что делать:

1. Откройте корень нужного проекта.
2. Создайте папку `.codex`, если ее нет.
3. Создайте файл `.codex/config.toml`.
4. Запишите туда project-specific defaults.

Например, логика уровня:

- работать в `workspace-write`;
- не давать лишнюю сеть;
- использовать конкретный approval policy.

Это правильное место для технического режима проекта.

### Кейс 3. Я хочу, чтобы только одна подпапка проекта имела свои правила написания

Что делать:

1. Перейдите в нужную подпапку.
2. Создайте там `AGENTS.md`.
3. Напишите только локальные правила этой зоны.

Например:

- в этой папке тексты только на русском;
- каждое действие в интерфейсе объяснять буквально;
- одна большая глава = один файл.

Не дублируйте туда заново весь мир.
Пишите только локальные уточнения.

### Кейс 4. Я хочу, чтобы Codex спрашивал перед конкретной внешней командой

Что делать:

1. Создайте `.rules` файл в `~/.codex/rules` или `.codex/rules`.
2. Задайте `prefix_rule()` с решением `prompt`.
3. Перезапустите Codex.
4. При желании проверьте правило через `codex execpolicy check`.

Это правильный путь, когда проблема не в стиле и не в памяти проекта, а именно в запуске команды вне sandbox.

### Кейс 5. Я хочу запретить опасный класс команд

Что делать:

1. Добавьте `.rules` с решением `forbidden`.
2. В `justification` напишите безопасную альтернативу.

Например, не просто:
"запрещено"

а:
"используй безопасную альтернативу X".

Так у пользователя и у агента остается понятный маршрут, а не тупик.

### Кейс 6. Я хочу временно запустить Codex в другом режиме только сегодня

Что делать:

1. Не переписывайте ради этого весь проектный конфиг.
2. Используйте CLI flags или `--config`.
3. Если такой режим нужен регулярно, оформите его как profile.

Логика здесь такая:

- разовый эксперимент должен жить как разовый override;
- постоянная привычка должна жить в конфиге.

### Кейс 7. Я хочу, чтобы компания запретила full access всем сотрудникам

Что делать:

1. Не пытайтесь решать это через пользовательские `AGENTS.md`.
2. Не пытайтесь надеяться на дисциплину каждого человека.
3. Делайте это через `requirements.toml`.

Потому что это уже не рекомендация, а организационный барьер.

### Кейс 8. Я не понимаю, куда писать правило, и все смешалось

Используйте этот быстрый фильтр.

Если правило звучит так:

- "пиши проще" -> `AGENTS.md`
- "работай в workspace-write" -> `config.toml`
- "на gh pr view всегда спрашивай" -> `.rules`
- "в компании нельзя danger-full-access" -> `requirements.toml`

Если вы ответили на этот вопрос честно, место для правила обычно становится очевидным.

## 10. Что не делать

### Не заменяйте `AGENTS.md` конфигом

`config.toml` не предназначен для редакционной памяти проекта.

### Не заменяйте `.rules` инструкциями

`.rules` нужен не для стиля, а для внешних команд и эскалаций.

### Не кладите секреты в инструкции

Ни в `AGENTS.md`, ни в проектные памятки, ни в примеры правил.

### Не делайте один гигантский файл "на все случаи жизни"

Лучше короткий глобальный слой плюс ясные локальные уточнения.

### Не включайте полный доступ по умолчанию без явной причины

Сначала должна быть задача, потом право, а не наоборот.

## 11. Глоссарий

**`AGENTS.md`**
Файл постоянных инструкций для Codex.

**`AGENTS.override.md`**
Файл, который имеет приоритет над `AGENTS.md` в той же папке.

**`.codex/config.toml`**
Локальный конфигурационный файл Codex для пользователя, проекта или подпапки.

**`sandbox_mode`**
Режим, который определяет технические границы действий агента.

**`approval_policy`**
Режим, который определяет, когда агент должен спрашивать подтверждение.

**`.rules`**
Файл с правилами `prefix_rule()` для команд вне sandbox.

**`prefix_rule()`**
Правило, которое описывает, что делать с командой по ее префиксу: разрешить, спросить или запретить.

**`requirements.toml`**
Организационный слой жестких ограничений, который пользователь не должен просто обойти локальным конфигом.

**`managed_config.toml`**
Организационный слой стартовых значений, которые применяются при запуске.

**Trusted project**
Проект, которому Codex доверяет настолько, чтобы загружать его `.codex/config.toml`.

**`project_doc_fallback_filenames`**
Настройка, которая позволяет искать альтернативные имена файлов для проектных инструкций.
