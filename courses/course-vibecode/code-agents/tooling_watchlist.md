## Цель

Отслеживать развитие ключевых инструментов для кода и агентных систем и раз в неделю получать краткий, немаркетинговый отчёт: какие **новые фичи** и **кейсы использования** реально могут повлиять на работу инженера.

**Целевая аудитория:** инженеры-практики. Отчёт должен отвечать на один вопрос: какая новая опция или фича появилась, как её использовать и в каком контексте? Приоритетные темы: автономия агентов и безопасность.

## Что считать «событием»

### 1. Новые фичи

Считать значимыми событиями:

- Запуск или крупное обновление фич: новые возможности в IDE, новые режимы агента/коворка, поддержка дополнительных языков/фреймворков, интеграции (GitHub, Jira и т.п.), серьёзные изменения UX.  
- Существенное улучшение качества: заметный рост скорости/надёжности, улучшение кода в конкретных бенчмарках, новые способности (например, рефакторинг больших баз, сложные миграции).
- **Новые CLI-флаги, ключи и режимы:** при упоминании — обязательно подробное описание: синтаксис, параметры, что конкретно меняет, пример команды. Для инструментов Приоритета 1 — развёрнутое объяснение.

**ВАЖНО: правило описания каждой фичи.** Не просто упоминать фичу — а сразу разворачивать объяснение по схеме:
  1. **Что это** — краткое техническое описание (синтаксис, параметры).
  2. **Зачем это нужно** — какую практическую проблему решает, в каком контексте полезно.
  3. **Пример использования** — конкретная команда или сценарий применения.
  
  Плохо: «--bare флаг для headless CI/CD скриптинга».
  Хорошо: «--bare — запуск без hooks/LSP/плагинов. Нужен для CI/CD: когда Claude Code работает в GitHub Actions, IDE-фичи не нужны и вызывают ошибки. Пример: `claude --bare -p "Review this PR" --output-format json`».

### 2. Кейсы использования

Считать значимыми кейсами:

- Реальные истории пользователей с подробным описанием:
  - **Проблема:** что нужно было решить, контекст задачи.
  - **Решение:** как именно применили инструмент, какие настройки/флаги использовали.
  - **Результат:** экономия времени, повышение качества, новый тип задач, который стал возможен. По возможности — с цифрами.
- Кейсы от продвинутых пользователей/команд (особенно в инженерных/прод‑контекстах), желательно с кодом, скриншотами, цифрами.

## Что игнорировать

- **Сделки, слияния, поглощения, скандалы** — никаких бизнес-новостей. Только технические фичи и кейсы использования.
- Чистый маркетинг и хайп: общие заявления без примеров, метрик, демо или кода.  
- Повторяющиеся анонсы, где нет новых технических деталей по сравнению с прошлой неделей.  
- Обсуждения, не относящиеся к практическому использованию инструментов (политика, корпоративные новости, общие «AI рассуждения» и т.п.).

## Формат недельного отчёта

**Формат файла: Markdown (.md)** — основной формат, удобный для обработки AI-инструментами.

Каждый отчёт должен содержать:

1. **Executive summary**  
   - 3–5 пунктов о том, какие технические изменения недели реально могут повлиять на работу инженера.

2. **Блок «По инструментам»**  
   Для каждого инструмента из списка (в порядке приоритета):
   - Краткий список новых фич (если были). **Для Claude Code и Codex:** при упоминании новых флагов/ключей/команд — подробное описание: синтаксис, параметры, что делает, пример использования.
   - Краткий список новых кейсов (если были) — в формате «проблема → решение → результат».
   - Оценка: полезно ли это прямо сейчас (да / нет / возможно позже) с пояснением в 1–2 предложения.

3. **Таблица сравнения**  
   Таблица с колонками:
   - Инструмент  
   - Новые фичи (кратко)  
   - Новые кейсы (кратко)  
   - Потенциальное влияние: низкое / среднее / высокое  
   - Нужны тесты на этой неделе: да / нет

4. **Рекомендации на неделю**  
   - Список не более чем из 3 конкретных действий: что именно протестировать, попробовать или поменять в рабочем процессе.

5. **Источники**
   - Список всех использованных источников с URL.

## Частота и первый запуск

- Базовая частота: **раз в неделю, по понедельникам**.  
- **Особое правило для первого запуска:** сразу после загрузки этого файла и создания задачи Computer должен немедленно сформировать первый отчёт по текущему состоянию и последним изменениям (без ожидания недели).
