# Twitter Product Filter — High‑Signal Products Only

## Цель

Отфильтровать шум в X (Twitter) и раз в неделю получать отчёт только о тех новых продуктах и фичах, которые потенциально стоят моего времени, с максимальным отсечением маркетинговой ерунды.

## Источники

- Есть список X‑аккаунтов (около 50, позже до 100), включающий:
  - Официальные аккаунты компаний и продуктов (AI‑инструменты, код‑ассистенты, агенты, workflow‑платформы и т.п.).
  - Аккаунты продвинутых пользователей и инженеров, которые делятся практическими кейсами.
- Аккаунты и твиты рассматриваются как **публичные**; доступ идёт через обычный веб‑просмотр (браузер Computer), не через мой личный аккаунт.

## Что собирать каждую неделю

За последнюю неделю для всех указанных аккаунтов:

1. Найти твиты, которые:
   - анонсируют новый продукт или важную фичу,  
   - рассказывают о реальном кейсе использования инструмента,  
   - содержат ссылки на лендинги, блоги, репозитории или демо‑страницы.

2. Для таких твитов обязательно:
   - перейти по ссылкам из твита,  
   - проанализировать лендинг/пост/репозиторий: понять, есть ли реальный продукт, API, демо, документация.

## Классификация сигналов

Каждый найденный продукт/фича должен быть отнесён к одной из категорий:

1. **Junk / Noise**  
   - Маркетинговый лендинг без продукта, демо или конкретики.  
   - Обещания без примеров, измеримых результатов или технических деталей.

2. **Interesting — Watchlist**  
   - Идея или подход выглядят любопытно.  
   - Есть какой‑то продукт/репозиторий/демо, но мало реальных кейсов или информации,  
   - либо доступ ограничен (closed beta), и нужно наблюдать, но не вкладывать время прямо сейчас.

3. **Actionable — Test Soon**  
   - Есть доступный продукт или API, понятные сценарии использования.  
   - Есть хотя бы один внятный кейс или демо с примером задачи и результатами.  
   - Есть вероятность реального прироста продуктивности или расширения возможностей по сравнению с текущим стеком.

Для каждого объекта указать:
- Причину классификации (1–3 предложения).  
- Категорию продукта (код‑ассистент, агент, workflow‑автоматизация, ресёрч‑инструмент, IDE, и т.п.).

## Что особенно важно

- Не верить твитам «на слово». Если нельзя проверить продукт (нет сайта, демо, доков) — помечать как **«unclear / manual review needed»**.  
- Отдавать приоритет инструментам, которые:
  - пересекаются с моим текущим стеком (Claude Code / Cowork, Codex, Cursor, Computer/CoWork, агенты и workflow‑платформы),  
  - или могут явно усилить его.

## Формат недельного отчёта

Каждый отчёт должен содержать:

1. **Summary**  
   - 5–10 продуктов/фич недели, имеющих наибольший потенциал (категория “Actionable — Test Soon”).  
   - Для каждого: название, кратко что это и почему стоит внимания.

2. **Таблица по всем найденным продуктам**  
   Колонки:
   - Название  
   - Основная ссылка  
   - Категория (код, агенты, workflow, ресёрч и т.п.)  
   - Классификация (Junk / Watchlist / Actionable / Unclear)  
   - Краткое объяснение (1–2 предложения)

3. **Рекомендации по действиям**  
   - Конкретный список, на что потратить 2–3 часа на неделе (какие продукты протестировать и что именно в них попробовать).

4. **История и эволюция**  
   - Если продукт уже появлялся ранее, отметить, что изменилось: новая фича, улучшение, новая инфа, изменение статуса (например, из Watchlist в Actionable).

## Частота и первый запуск

- Базовая частота: **раз в неделю, по понедельникам**.  
- **Особое правило для первого запуска:** сразу после загрузки этого файла и создания задачи Computer должен немедленно собрать первый отчёт по текущей неделе, не ожидая следующего понедельника.
